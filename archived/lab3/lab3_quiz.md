# Question 1: Tiled Matrix Multiplication Memory Use

## 문제

MP3에서 사용하는 타일드 행렬 곱셈 커널에 대해, 아래 조건에서 **전역 메모리에서 읽는 바이트 수(A)** 와 **전역 메모리에 쓰는 바이트 수(B)** 를 구하시오.

**주어진 값**

| 변수 | 값 |
|------|-----|
| numARows | 63 |
| numAColumns | 41 |
| numBRows | 41 |
| numBColumns | 55 |
| numCRows | 63 |
| numCColumns | 55 |
| 데이터 타입 | float (4 bytes) |
| 타일 크기 | 8 × 8 |

- **A:** 커널에 의해 전역 메모리에서 읽히는 바이트 수 (integer)
- **B:** 커널에 의해 전역 메모리에 쓰여지는 바이트 수 (integer)

---

## 풀이 과정

### 1. Grid / Block 구조

- **Block:** 8×8 스레드 (타일 크기와 동일)
- **Grid:** 출력 행렬 C의 크기를 8×8 타일로 덮도록 올림
  - Grid x = ⌈numCColumns / 8⌉ = ⌈55/8⌉ = **7**
  - Grid y = ⌈numCRows / 8⌉ = ⌈63/8⌉ = **8**
- **총 블록 수:** 7 × 8 = **56 blocks**

### 2. 타일 개수 (k 방향)

- 한 블록이 C의 한 타일(8×8)을 계산할 때, 내적을 위해 A의 열·B의 행 방향으로 나누어 읽는다.
- 타일 개수: `numTiles = (numAColumns - 1) / 8 + 1 = (41-1)/8 + 1 = 6`

### 3. A: 전역 메모리 읽기 (Bytes)

**해석:** “전역 메모리에서 읽히는 바이트”를 **실제로 읽는 데이터 양(유효 로드)**으로 센다.  
즉, 같은 원소를 여러 블록이 읽더라도, 그 원소가 메모리에서 한 번 읽혀 나갈 때마다 1번으로 센다.  
(또는 “각 원소가 참조되는 총 횟수 × 4 bytes”로, 중복 포함.)

- **행렬 A (63×41)**  
  - 블록 (bx, by)는 A의 **행** by×8 ~ by×8+7 을 사용한다.  
  - 같은 **block row** (같은 by)에 있는 블록들은 **같은 8개 행의 A**를 사용한다.  
  - block row 개수 = 8 이고, 각 block row당 블록 수 = 7 이므로,  
    **A의 각 행**은 그 행을 포함하는 block row의 7개 블록에서 각각 읽힌다.  
  - 따라서 A에서 읽히는 원소 수 = 63행 × 41열 × 7 = **18,081**  
  - A 읽기 바이트 = 18081 × 4 = **72,324 bytes**

- **행렬 B (41×55)**  
  - 블록 (bx, by)는 B의 **열** bx×8 ~ bx×8+7 을 사용한다.  
  - 같은 **block column** (같은 bx)에 있는 블록들은 **같은 8개 열의 B**를 사용한다.  
  - block column 개수 = 7, 각 block column당 블록 수 = 8 이므로,  
    **B의 각 열**은 그 열을 포함하는 block column의 8개 블록에서 각각 읽힌다.  
  - 따라서 B에서 읽히는 원소 수 = 41행 × 55열 × 8 = **18,040**  
  - B 읽기 바이트 = 18040 × 4 = **72,160 bytes**

- **전체 읽기:** 72324 + 72160 = **144,484 bytes**

**답 A: 144484 (Bytes)**

### 4. B: 전역 메모리 쓰기 (Bytes)

- **쓰기 조건:** 커널에서 `if (row < numCRows && col < numCColumns)` 일 때만 `C[row][col]`에 쓴다.
- 따라서 전역 메모리에 실제로 쓰이는 원소 수는 **C의 유효 원소 개수**와 같다.
- **쓰기 개수:** numCRows × numCColumns = 63 × 55 = **3465** elements  
- **쓰기 바이트:** 3465 × 4 bytes = **13860 bytes**

**답 B: 13860 (Bytes)**

---

## 정리

| 질문 | 답 |
|------|-----|
| **A. 전역 메모리에서 읽히는 바이트** | **144484** |
| **B. 전역 메모리에 쓰여지는 바이트** | **13860** |

### 요약 식

- **읽기 (유효 데이터 기준):**  
  - A: `numARows × numAColumns × (Grid x) × 4` = 63×41×7×4 = 72324  
  - B: `numBRows × numBColumns × (Grid y) × 4` = 41×55×8×4 = 72160  
  - 합계: **144484**
- **쓰기:**  
  `numCRows × numCColumns × 4` = 63×55×4 = **13860**

# Question 2: Programming with Barriers

## 문제

아래 커널 코드 조각은 “일부 스레드만 일을 하고, 서로 동기화한 뒤, 그 다음 일을 한다”는 의도로 작성되었다.

```c
// some work for all threads
if (threadIdx.x < totalSize) {
  // do some work
  __syncthreads ();
  // do some more work
} else {
  __syncthreads ();
}
// some more work for all threads
```

**질문:** 이 코드의 문제점과 수정 방법을 가장 잘 설명한 보기를 고르시오.

---

## 선택지

1. **트릭 문제이다. 주석으로 대체된 코드(생략된 부분)에 버그가 있으므로, 그 부분을 수정해야 한다.**
2. **`__syncthreads`는 블록 내 일부 스레드만 호출하면 안 된다. 두 그룹을 서로 다른 스레드 블록(또는 서로 다른 커널 실행)으로 나누어야 한다.**
3. **블록 내 모든 스레드가 동일한 “정적” `__syncthreads` 호출—즉, 코드 상 동일한 라인—을 실행해야 하며, 호출 “횟수”만 같으면 부족하다. 따라서 동일한 조건문을 두 번 사용하고, `__syncthreads`는 조건문 밖에서 실행해야 한다.**
4. **`totalSize`보다 큰 인덱스를 가진 스레드는 할 일이 없으므로, 조건문 밖의 추가 코드를 then 블록 안으로 옮기면 된다.**
5. **조건을 만족하지 않는 스레드는 할 일이 없으므로, 그들에게 `__syncthreads`를 강제하는 것은 커널만 느리게 한다. else 절을 삭제하면 된다.**

---

## 풀이

### 문제점

- `__syncthreads()`는 **블록 내 모든 스레드**가 **같은 정적 호출 지점(같은 라인)**에 도달해야 한다.
- 위 코드에서는  
  - `threadIdx.x < totalSize` 인 스레드는 **if 블록 안의** `__syncthreads()`에 도달하고,  
  - 그렇지 않은 스레드는 **else 블록 안의** `__syncthreads()`에 도달한다.  
- 즉, 서로 **다른 정적 호출**에서 대기하게 되어, 한쪽은 if의 barrier에서, 다른 쪽은 else의 barrier에서 영원히 기다리게 되고 **데드락**이 발생한다.

### 올바른 수정

- “같은 횟수”가 아니라 **같은 라인의** `__syncthreads()`를 모두가 실행해야 하므로, barrier를 **조건문 밖**으로 빼야 한다.
- 예: 동일한 조건을 두 번 쓰고, 그 사이에 `__syncthreads()`를 두 번(각각 조건문 밖에서) 호출한다.

```c
// some work for all threads
if (threadIdx.x < totalSize) {
  // do some work
}
__syncthreads();   // 모든 스레드가 같은 호출에 도달
if (threadIdx.x < totalSize) {
  // do some more work
}
__syncthreads();   // 모든 스레드가 같은 호출에 도달
// some more work for all threads
```

### 정답

**3번.**  
블록 내 스레드는 **같은 정적 `__syncthreads` 호출(같은 코드 라인)**을 실행해야 하며, 호출 횟수만 같으면 안 된다. 동일한 조건문을 두 번 사용하고, `__syncthreads`는 조건문 밖에서 실행해야 한다.

# Question 3: Tiled Matrix Multiplication Computation Use

## 문제

MP3 타일 행렬곱 커널에서 아래와 같이 주어진 경우, 부동소수점 연산(덧셈·곱셈 등) 횟수를 구하시오.

**주어진 값**

| 변수 | 값 |
|------|-----|
| numARows | 147 |
| numAColumns | 91 |
| numBRows | 91 |
| numBColumns | 113 |
| numCRows | 147 |
| numCColumns | 113 |
| 타일 크기 | 32 × 32 |

- **A:** 출력 행렬 C의 원소를 담당하는 스레드만 부동소수점 연산을 수행한다고 할 때, 실행되는 부동소수점 연산의 총 개수 (integer)
- **B:** 실행되는 모든 스레드가 부동소수점 연산을 수행한다고 할 때, 실행되는 부동소수점 연산의 총 개수 (integer)

---

## 풀이 과정

### 공통: 원소당 연산 수

- C의 한 원소 `C[row][col]`는 내적  
  `sum = Σ_k A[row][k] * B[k][col]`  
  (k = 0 .. numAColumns−1 = 91−1 → **91번** 반복)
- 반복당: 곱셈 1회, 덧셈 1회 → **2 FLOPs**
- 따라서 **유효한 출력 원소 하나당** FLOPs = 91 × 2 = **182**

---

### A: C를 담당하는 스레드만 연산할 때

- 유효한 C 원소 수 = numCRows × numCColumns = **147 × 113 = 16,611**
- 이 원소들만 위 내적을 수행하므로  
  **총 FLOPs = 16,611 × 182 = 3,023,202**

**답 A: 3023202**

---

### B: 런치된 모든 스레드가 연산할 때

- **그리드:** 타일 32×32 기준  
  - Grid x = ⌈113/32⌉ = **4**  
  - Grid y = ⌈147/32⌉ = **5**  
  - 블록 수 = 4 × 5 = **20**
- **스레드 수:** 20 × (32×32) = 20 × 1024 = **20,480**
- **타일 수 (k 방향):** (91−1)/32 + 1 = **3**
- **스레드당 연산:**  
  - 타일마다 내부 루프 k = 0..31 → 32번, 반복당 2 FLOPs → 타일당 64 FLOPs  
  - 스레드당 = 3 × 64 = **192 FLOPs**
- **총 FLOPs:** 20,480 × 192 = **3,932,160**

**답 B: 3932160**

---

## 정리

| 질문 | 답 |
|------|-----|
| **A. C 담당 스레드만 연산** | **3023202** |
| **B. 모든 스레드 연산** | **3932160** |

# Question 4: Shared Memory Variables

## 문제

커널이 **235개의 스레드 블록**으로 실행되며, 각 블록은 **626개의 스레드**를 가진다. 이 커널에 **공유 메모리 변수 X**가 선언되어 있다.

**질문:** 커널 실행 동안 생성되는 X의 **버전(복사본) 개수**는? (integer, versions)

---

## 풀이

- CUDA에서 `__shared__` 변수는 **블록 단위(block-wide)** 로 할당된다.
- 즉, **각 스레드 블록마다** 그 블록만의 공유 메모리 영역이 생기고, 그 안에 X가 하나씩 존재한다.
- 스레드 개수(626)는 블록당 스레드 수일 뿐, 공유 메모리 “버전” 수와는 무관하다.

**답: 235**

# Question 5: Programming with Barriers II

## 문제

Lab 3 커널 구현에서 `__syncthreads()`의 **목적**은 무엇인가? 맞는 것을 모두 고르시오.

**(a)** 공유 메모리 타일이 로드된 뒤에야 FP 연산이 시작되도록 한다.  
**(b)** 다음 for 반복에서 타일을 덮어쓰기 전에, 현재 공유 메모리 타일이 완전히 사용되도록 한다.  
**(c)** 스레드가 올바른 전역 메모리 위치를 방문하고, 그 값을 전역 메모리에서 공유 메모리로 복사할 수 있도록 한다.  
**(d)** halo cell을 다루는 스레드가 행렬 값을 공유 메모리에 복사하지 못하도록 한다.  
**(e)** 위 보기 중 맞는 것이 없다.

---

## 풀이 (Lab 3 타일 커널 기준)

커널 구조 요약:

```text
for (tile = 0 .. numTiles) {
  // A_s, B_s 타일 로드 (각 스레드가 전역 메모리 → 공유 메모리)
  __syncthreads();   // (1)
  for (k = 0 .. TILE_WIDTH)
    sum += A_s[...] * B_s[...];  // FP 연산
  __syncthreads();   // (2)
}
```

- **(a) 맞음.**  
  (1)의 barrier는 “타일 로드”가 **모든 스레드에 의해 끝난 뒤**에만 다음 FP 연산이 시작되게 한다. 그래서 shared tile이 먼저 채워진 다음에 곱·합이 이루어진다.

- **(b) 맞음.**  
  (2)의 barrier는 “현재 타일을 읽는 FP 연산”이 **모든 스레드에 의해 끝난 뒤**에만 다음 반복으로 넘어가게 한다. 그래서 다음 반복에서 `A_s`, `B_s`를 덮어쓰기 전에, 현재 타일이 완전히 활용된다.

- **(c) 틀림.**  
  올바른 전역 메모리 위치 계산과 복사는 `row`, `col`, `tile`과 인덱스 식으로 이루어진다. `__syncthreads()`는 “언제” 동기화할지만 정할 뿐, “어디를 읽을지”를 보장하지 않는다.

- **(d) 틀림.**  
  경계(halo) 처리에서는 조건문으로 0을 넣거나 유효한 값만 복사한다. barrier는 “halo 스레드가 복사하지 못하게” 하는 역할이 아니며, 그런 의도로 쓰이지도 않는다.

- **(e) 틀림.**  
  (a), (b)가 올바른 설명이다.

### 정답

**(a), (b)** 를 선택한다.
